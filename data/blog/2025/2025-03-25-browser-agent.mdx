- LLM 下的 Automation
  - 今天 在 Agent 的 语境下， 我们 需要 重新 思考 Automation 的 定义， 扩展 AI 对 电脑手机和浏览器等 设备 的 控制
  - 过去很多年， 我们一直在讨论人机交互， 设备和软件制造出来， 是为人类服务的（当然， 也有很多软件和设备， 是 为 机器 服务的）
  - 过去很长时间， 我们制造了大量的软件， 适配人类使用的方式来设计。
- 但是， 今天， 我们 需要 重新 思考 这个 问题， 我们需要 重新 定义 人机交互， 我们需要 重新 定义 设备和软件制造的目的。
  - 在 LLM Agent 的语境下， AI 变得比历史上任何时候都更加强大， 我们再也不会面对当年老罗在发布会上的尴尬。
  - 在 LLM 的帮助下， 结合 tool call， AI 能够更准确的调用 电脑手机和浏览器等 设备 的 能力， 从而 完成 更加 复杂的 任务。Siri 也不会因为识别问题， 而 无法 完成 用户的 需求。

这里面一个很大的场景， 是我们需要将曾经花了好几十年制造的设备和 UI， 让 AI 来 控制。 当然， 在将来，我们可能会重新思考软件开发的问题， 以后也许软件更多是被 AI 而不是人类来使用， 在那个时候，软件的核心应该是为 AI 提供数据， 而不是为用户做展示。

但是， 今天， 我们还是需要想办法，让 AI 更加高效准确地使用这些设备。
这就像， 今天出厂的汽车都带有行车记录仪， 但是，老车也想要使用这个功能， 怎么办？ 我们只能去生成独立的行车记录仪，通过已有的历史遗留的接口，连接到汽车上。

在所有的用户软件中，最大的一类应该就是浏览器了。 本来， 我们处于移动时代， 更多使用的是移动端端 Android 和 iOS 软件， 但是，因为可能不能更好地在移动设备上使用自动化技术去控制，所以，我们改为使用浏览器。 当然也可能是因为， 浏览器是方便放在服务端去控制， 所以，我们才使用浏览器。

在过去的 20 年中， 我们的web技术以及 web ui 自动化测试技术， 已经非常成熟。

这里我们 产生了通过 AI 控制浏览器的需求。 这里面有两项核心的技术

1. 基于 LLM 的 推理和代码生成
2. web 自动化测试技术

基本逻辑， 我们让LLM 生成 一个 浏览器 的 自动化测试脚本， 然后让LLM 按需 执行 这个 脚本， 从而 控制 浏览器。

当然， 我们也可以 通过 工具调用 的方式， 让 LLM 调用 web driver 的函数控制浏览器， 但是，这种方式有一个很大的问题，

1. 对于复杂任务力不从心，需要很多次 LLM 交互， 也不一定能很好解决。
2. web driver 等的接口非常多，在实现上， 就比较困难。

一种更好的方式， 是直接让 LLM 为任务生成 对应的自动化脚本（python 或者 javascript）， 然后 执行 这个 脚本， 从而 控制 浏览器。
而这些脚本在网上有大量的代码， 前沿的LLM 已经学习了大量的浏览器自动化相关代码， 从而 不难生成对应的脚本代码。

## 未来

现状是 AI 时代已经来临， 但是我们的软件和设备很多还是处于以前的人机交互， 我们需要让 AI 学会 适配使用这些设备。

未来， 我们的软件很大程度上将由 AI 来开发，必然也会是 AI native 的， 更方便 AI 去使用。
比如基于 API 的接口， 或者基于 agent 协议的接口。

目前社区已经由一些团队在探索 agent 的协议， 比如
